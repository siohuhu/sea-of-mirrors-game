### 曾解決的問題紀錄

**iOS 音效 Bug**
- 問題：iOS 上背景音樂無法自動循環播放
- 原因：iOS Safari 限制 `loop` 屬性的自動播放行為
- 解法：移除 `loop` 屬性，改用 `ended` 事件監聽器手動重播
```javascript
audio.addEventListener('ended', () => { audio.play(); });
```

**字型載入問題**
- 問題：自訂字型在部分瀏覽器 / 裝置上無法正常顯示
- 排查：重複 `@font-face` 宣告衝突、font-family 名稱不一致、Safari Lockdown Mode 封鎖 Google Fonts
- 解法：統一 `@font-face` 宣告、確認名稱一致、改為本地端托管

**iOS 互動音效**
- 開始按鈕 hover 音效改為 touch 事件觸發（iOS 不支援純 CSS hover）
- 封面到測驗頁 fade 轉場：經多次微調 timing，確保音效與動畫同步

## 9. 維護紀錄

### 2026-06-17　Firebase Token 重新申請

- **項目**：sea-of-mirrors Firebase 存取 Token
- **動作**：重新申請並更新 Token
- **原因**：sea-of-mirrors 測試頁原本透過終端機下指令推送，改由 GitHub Desktop 推送後，發現 Firebase Token 曾外流至 Git 紀錄中，因此立即重新申請新 Token
- **影響範圍**：sea-of-mirrors 測試頁與 Firestore 的連線授權
- **處置**：
  - 舊 Token 立即作廢，重新申請新 Token ✅
  - 使用 BFG Repo Cleaner 從 Git 完整歷史中徹底清除外流檔案 ✅
  - `git reflog expire` + `git gc --prune=now` 清理殘留參照 ✅
  - `git push origin main --force` 強制更新遠端 repo ✅
  - 確認 `.gitignore` 防線已設定（`*firebase-adminsdk*.json` / `*secret*.json`）✅
  - 新 Token 不進 Git，存放於以下三個位置 ✅
    - 外接碟：`SiohuhuDev/Dev/`
    - 本機：`~/Dev/`
    - Google Drive 雲端備份
- **教訓**：
  - `.gitignore` 只能擋「還沒被 commit 過」的檔案，已追蹤過的檔案需先用 `git rm --cached` 移除追蹤再加入 `.gitignore`
  - 從終端機切換到 GitHub Desktop 等新工具時，務必先驗證 `.gitignore` 正確排除所有憑證檔案，再執行第一次推送
  - 憑證外流第一優先：讓舊憑證失效，比清除 Git 歷史更急

## 11. 防盜版設計策略（2026-06-22）

### 背景
前作 App Store 發行第 3 天即遭盜版，整個遊戲被破解並流出至網路，原因不明。
本作設計目標：**即使被盜，盜版版本也會因遊戲機制變得沒有意義**，而非單純加強技術防破解。

---

### 核心策略：讓盜版「有得玩但沒有靈魂」

**1. 核心體驗綁定 Firebase 雲端**
- 個性測驗結果、旅人原型、社群分享功能需連線才能完整體驗
- 離線 / 盜版版本可以跑起來，但資料不存在、分享無效、社群感消失
- 每日抽牌、里程碑紀錄綁定帳號，盜版玩家進度無法保留

**2. 內容綁定帳號進度（雲端存檔）**
- 22 張大阿爾卡那解鎖進度、旅人故事推進存於雲端，不存本地
- 盜版版本每次從零開始，沒有累積感
- 對塔羅 × idle 類型特別有效（idle 核心吸引力就是累積感）

**3. 社群功能作為核心體驗**
- 分享旅人原型、LINE 分享卡、`navigator.share` 系統——盜版版本天生缺失
- 看到其他人的鏡中世界、與朋友比對原型，這些體驗無法被複製

**4. 定期內容更新**
- 節慶活動、限定塔羅牌組合、季節性世界變化
- 盜版版本是靜止快照，正版用戶持續有新內容
- 參考 Tabi Kaeru 的核心策略

---

### 技術層面輔助防線

- **iOS 平台本身**：沙盒機制讓 IPA 就算被取出，很多功能也無法正常運作，比 Android 難破解
- **PC 版注意**：PC 端相對容易被破解，PC 版的核心內容同樣需要雲端化
- **不建議投入太多資源的方向**：混淆程式碼、加殼、反除錯——只增加破解難度，無法真正防禦，且拖慢開發速度

---

### Sea of Mirrors 防盜版核心公式

```
雲端進度（Firebase）
+ 社群分享體驗（LINE、navigator.share）
+ 定期內容更新
= 盜版版本是殘缺的靜態殼，正版才有完整的「鏡之海」體驗
```

## 12. 核心機制與資安防禦規範（2026-06-25）

### 設計原則
破解版能啟動、能散步、能看風景，但失去完整世界體驗。
不是讓盜版「無法執行」，而是讓盜版「沒有靈魂」。

---

### 四層防線架構

#### 第一層：遊戲本體可離線（客戶端）
玩家沒有網路仍可遊玩，保障基本體驗與 App Store 評價：
- 探索
- 閱讀
- 散步
- 收集普通物件

> ⚠️ App Store 評價容易受影響，離線體驗必須完整且有品質。

---

#### 第二層：每日內容雲端生成（Firebase Cloud Functions）
每日內容全部由伺服器產生，無法在本地偽造：
- 今日鏡像
- 今日來信
- 今日塔羅
- 今日旅人
- 今日動物

即使破解版修改以下項目，也無法控制結果：
- 本地時間
- 存檔
- 記憶體

---

#### 第三層：收藏價值放雲端（帳號綁定）
所有稀有內容綁定帳號，破解版無法取得：
- 稀有卡牌
- 傳說故事
- 特殊插畫
- 節慶事件

破解版可以玩，但無法：
- 收藏
- 同步
- 取得稀有內容

---

#### 第四層：世界共同事件（伺服器決定）
全球玩家共享的世界事件，全部由伺服器控制：
- 霧海開啟
- 白鹿出現
- 鏡鳥遷徙
- 流星夜

就算有人破解 IPA、反編譯 Unity、修改 Assembly-CSharp，
仍然只能玩到**離線殼**，而不是完整世界。

---

### 客戶端打包規範

**客戶端只包含：**
```
Card ID
Card Type
Card Placeholder
```

**禁止寫在 Unity 的邏輯：**
- 稀有獎勵判定
- 收藏判定
- 任何核心收藏系統

**可以寫在 Unity 的邏輯（不重要的視覺表現）：**
- 鳥飛哪裡
- 風吹方向
- 粒子效果
- NPC 閒逛

---

### 防線效果總結

| 破解行為 | 能得到什麼 | 失去什麼 |
|---|---|---|
| 破解 IPA | 離線殼 | 每日內容、收藏、雲端事件、社群同步 |
| 反編譯 Unity | 客戶端視覺邏輯 | 核心收藏系統、稀有獎勵判定 |
| 修改 Assembly-CSharp | NPC 行為、粒子效果 | 所有雲端綁定功能 |
| 修改本地時間／存檔 | 無效操作 | Firebase 雲端結果無法被影響 |

## 15. 伴寵系統設計核心（2026-06-25）

### 設計哲學：繫伴

靈感來源：《小王子》與狐狸的「馴養」概念。
不是玩家「擁有」寵物，而是玩家與伴寵之間建立**承諾的責任**。

> 「你要對你所馴養的負責。」

- 伴寵不是玩家選擇的，是玩家的**元素屬性**決定哪隻伴寵找上你
- 不是你選擇它，是它選擇你
- 承諾是單向的、永久的，不能丟棄或替換

### 機制影響

- 伴寵不能被替換（承諾永久）
- 玩家每日行為（任務、回應來信）影響伴寵狀態
- 長時間不登入：伴寵不消失，但有變化（等待、思念）
- 伴寵每天帶回世界訊息，與第二層「今日動物」雲端系統連動

---

### 帳號系統需求

承諾必須能**跨裝置保留**——玩家換手機，伴寵還在等他。
匿名驗證無法支撐此核心概念，帳號系統是必要設計，不是可選項。

### 個資考量與技術方案

**採用方向：Apple Sign In 匿名模式**

玩家選擇不公開真實 email，Apple 產生隨機中繼 ID：
- Firebase 只儲存匿名 `userID`，不含任何個人資料
- 跨裝置一致，換手機伴寵仍在
- 符合 GDPR、台灣個資法
- App Store 隱私標籤可填「不收集個人資料」

**關於失去**

玩家主動撤銷 Apple Sign In 授權 → 匿名 ID 消失 → 伴寵永遠失去，無法找回。

這不是缺陷，是設計核心的延伸：

> 承諾需要雙方維持。玩家主動切斷，就是主動放棄承諾。
> 失去就沒了——正因如此，更需要珍惜。

「失去的不可逆」強化了繫伴的重量，讓承諾更真實。

- 帳號設計、隱私政策、App Store 隱私標籤待與合作運營商討論

## 16. 霧的機制設計（2026-06-25）

### 霧的定位
霧不是第五個玩家屬性，而是**超越元素之上的機制層**。

```
火 / 水 / 風 / 土  ←  玩家屬性（四選一，固定）
        ↓
       霧          ←  遊戲機制層（命運、機會、指引）
```

白鹿不屬於任何玩家，在關鍵時刻出現引路，作用於所有玩家。
對應第四層防線「世界共同事件」：白鹿出現為全服事件，伺服器決定。

---

### 霧的觸發機制（三層權重）

**第一層：節氣預告（黏著機制）**
- 節氣前一週放出預告訊息
- 製造期待感、提醒玩家回來、建立世界節律感

**第二層：玩家行為累積（參與感）**
- 達到一定登入次數解鎖觸發資格
- 門檻不高，獎勵回訪，不懲罰輕度玩家

**第三層：伺服器隨機（命運感）**
- 有資格的玩家中，伺服器決定實際觸發時機
- 永遠無法被計算，保留命運的不可預測性

```
三層同時成立 → NPC 跳出預言
```

## 17. 敘事設計原則（2026-06-25）

### 無聲世界

**整個遊戲不說話，只有文字。其餘由玩家心理填補。**

- 無語音、無旁白、無對話配音
- 所有敘事透過文字呈現
- 聲音設計只有音樂與環境音效
- 留白給玩家自己的想像與投射

### 設計意義
- 玩家成為故事的共同創作者，不是被動接收者
- 每個人讀到的「鏡之海」都不完全一樣
- 與塔羅牌的本質吻合——牌面沉默，意義由解讀者賦予
- 降低本地化成本（無配音，只需翻譯文字）

### 對伴寵系統的影響
- 伴寵不說話，只有動作與狀態變化
- 玩家自行解讀伴寵的情緒與意圖
- 繫伴的情感連結由玩家主動建立，而非遊戲強迫
